# `camera_aim` PID 调参流程文档

## 1. 目的

本文用于指导 `camera_aim` 控制端在 Gazebo 和真机环境下进行 PID 调参。

当前 `camera_aim` 控制链路为：

`err_x / err_y -> pid_cam_aim_ -> body_delta -> max_step 限幅 -> ENU 增量 -> /mavros/setpoint_position/local`

调参目标不是让飞机“动作越大越好”，而是：

- 能稳定朝目标中心收敛
- 接近中心时不过冲、不来回抖
- 能连续满足稳定条件并触发 `capture_ready`
- 视觉短时抖动时不过度放大机体运动

---

## 2. 当前实现涉及的关键参数

### 2.1 已参数化的外围参数

这些参数已经可以在 YAML 中直接调整：

- `camera_aim_max_step`
  - 单周期位置增量限幅，单位 `m`
  - 太大容易冲和抖
  - 太小会导致响应慢

- `camera_aim_stable_cycles`
  - 连续多少个控制周期都满足误差阈值，才认为“对准成功”
  - 太大时不容易触发 `capture_ready`
  - 太小时容易误判“对准成功”

- `camera_aim_target_timeout_s`
  - 当前目标多久没更新就判定为丢失

- `camera_aim_record_result_timeout_s`
  - 发出 `capture_ready` 后，等待视觉端 `record_result` 的超时时间

- `camera_aim_scan_point_timeout_s`
  - 单个扫描点总处理超时时间

- mission 里的 `camera_aim tolerance`
  - 判断 `err_x / err_y` 是否进入稳定区的阈值

### 2.2 当前 PID 参数

当前代码中的 PID 参数已经支持通过 YAML / ROS 参数配置，对应代码位置：

- [action_executor.hpp](/home/bosen/drone_ws/src/drone_control/include/drone_control/action_executor.hpp:1446)

默认值为：

```cpp
Vec3dPID pid_cam_aim_{0.001, 0.0, 0.001};
```

对应关系：

- `P = 0.001`
- `I = 0.0`
- `D = 0.001`

对应 YAML 参数名为：

- `camera_aim_pid_p`
- `camera_aim_pid_i`
- `camera_aim_pid_d`

因此后续调参时，优先直接修改 YAML 并重新启动节点，不需要再改 C++ 代码。

## 3. 调参前先理解现象

调参时，不要只看“有没有触发 `capture_ready`”，而要重点观察以下现象：

- 目标偏差大时，飞机是否明显朝正确方向修正
- 接近中心时，误差是否快速减小
- 接近中心后是否来回穿越中心点
- 机体是否出现高频小抖动
- 误差已经很小，但是否长时间就是触发不了稳定完成

建议同时观察：

- `/mission_status`
- `/k230/animals/targets`
- `/mavros/local_position/pose`
- `mission_controller_node` 日志

---

## 4. 调参顺序

推荐顺序：

1. 先调外围参数
2. 只调 `P`
3. 再少量加 `D`
4. 最后才考虑是否需要 `I`

不要一开始同时改 `P / I / D / max_step / tolerance`，否则无法判断到底是谁起了作用。

---

## 5. 第一步：先固定外围参数

在开始调 PID 前，先把外围参数设成一组保守值。

建议初始值：

```yaml
camera_aim_max_step: 0.02
camera_aim_stable_cycles: 8
camera_aim_target_timeout_s: 0.5
camera_aim_record_result_timeout_s: 5.0
camera_aim_scan_point_timeout_s: 15.0
```

mission 中 `camera_aim` 动作的 `tolerance` 建议先保持一个偏宽松但不是特别大的值，例如：

```yaml
tolerance: 8.0
```

说明：

- `max_step` 小一点，先保证动作保守，不容易大幅摆动
- `stable_cycles` 先小一点，方便快速观察“是否已经基本对准”
- `scan_point_timeout` 在真机建议先保守设短，避免长时间悬停等待

---

## 6. 第二步：先调 P

### 6.1 P 的作用

`P` 决定“误差一出现，立刻给多大修正量”。

### 6.2 现象判断

`P` 太小：

- 目标偏差明明很大，但机体修正很慢
- `err_x / err_y` 下降太慢
- 很久都靠不近中心区域

`P` 合适：

- 能明显、稳定地朝中心收敛
- 接近中心时没有大幅摆动

`P` 太大：

- 一接近中心就过冲
- 左右来回穿越目标中心
- 很难连续满足稳定条件

### 6.3 实操方法

建议：

1. 保持 `I = 0`
2. 保持 `D` 很小或先不动
3. 每次只增加 `P` 的 `20% ~ 30%`
4. 每改一次，只测一个简单单目标场景

示例调法：

```text
P: 0.0010 -> 0.0012 -> 0.0015 -> 0.0018
```

一旦出现明显来回摆动，就说明 `P` 可能已经偏大了，应回退一档。

---

## 7. 第三步：再调 D

### 7.1 D 的作用

`D` 用于抑制接近目标中心时的过冲和摆动。

### 7.2 什么时候该加 D

如果出现下面这种情况，就可以开始加 `D`：

- `P` 已经足够让目标快速靠近中心
- 但接近中心时会来回摆
- 或误差在中心附近来回震荡

### 7.3 现象判断

`D` 太小：

- 收敛时仍然容易冲过头
- 误差围绕中心反复摆动

`D` 合适：

- 接近中心时明显更稳
- 更容易连续满足稳定判定

`D` 太大：

- 机体动作变钝
- 误差明明不小，但修正不积极
- 有时会出现细碎抖动

### 7.4 实操方法

在 `P` 基本定住后，每次小步增加 `D`。

示例：

```text
D: 0.0010 -> 0.0012 -> 0.0015
```

如果你发现“更稳了”，说明方向是对的。
如果发现“动作发僵、反应慢、甚至小抖”，说明 `D` 加多了。

---

## 8. 第四步：I 一般最后才考虑

### 8.1 为什么不要一开始就加 I

`camera_aim` 是视觉对准场景，不是长期定点悬停抗稳态偏差的典型速度环/位置环。

很多情况下：

- `P + D` 就足够

### 8.2 只有什么情况下才考虑加 I

只有当下面现象长期稳定出现时，才考虑少量加入 `I`：

- 误差能明显减小
- 但总是卡在一个很小的残差附近
- 长时间就是不进稳定阈值
- 且这个残差不是随机抖动，而是固定偏一侧

### 8.3 风险

`I` 太大容易导致：

- 误差累积过量
- 接近中心后继续冲
- 视觉误差一波动，机体出现拖尾感

因此建议：

- 没明确必要就保持 `I = 0`
- 如果要加，只加非常小的量

## 9. `max_step` 和 PID 的关系

即使 PID 输出很大，最终仍然会被 `camera_aim_max_step` 限幅。

所以会出现两种典型误判：

### 9.1 看起来 P 很小，其实是 `max_step` 太小

现象：

- 误差很大时，机体每次只挪一点点
- 无论怎么加 `P`，动作变化都不明显

原因：

- PID 输出已经被 `max_step` 截住了

### 9.2 看起来 P 很大，其实是 `max_step` 太大

现象：

- 一旦开始修正，动作就很猛
- 接近中心时明显容易冲过头

所以调 PID 时，一定要同时关注 `camera_aim_max_step`。

实战建议：

- Gazebo 初期：`0.02 ~ 0.03`
- 真机初期：`0.02`
- 确认稳定后再酌情加到 `0.03 ~ 0.05`

---

## 10. `tolerance` 和 `stable_cycles` 的关系

很多时候不是 PID 没调好，而是“稳定判定太严格”。

### 10.1 tolerance 太小

现象：

- 误差已经很接近中心
- 但就是很难连续满足条件
- `aim_close_count_` 经常刚累计几次又清零

### 10.2 stable_cycles 太大

现象：

- 已经进入小误差区
- 但要等很久才触发 `capture_ready`

### 10.3 建议

联调初期建议：

- `tolerance` 先宽一点
- `stable_cycles` 先小一点

等闭环已经稳定后，再逐步收紧要求。

## 12. 推荐调参流程

### 阶段 A：Gazebo 初调

目标：

- 先把“能稳定收敛并进入 `capture_ready`”这件事跑通

建议：

- 只测单目标
- `stable_cycles` 先设小
- `max_step` 先保守
- `I = 0`

### 阶段 B：Gazebo 收敛优化

目标：

- 减少摆动
- 提高触发稳定性的成功率

建议：

- 固定 `P`
- 小步加 `D`
- 视情况收紧 `tolerance`

### 阶段 C：真机保守迁移

目标：

- 把 Gazebo 里稳定的参数迁移到真机

建议：

- 先把 `max_step` 再减小一档
- 先测单目标
- 先低空短悬停
- 先验证超时退出链路

### 阶段 D：真机细调

目标：

- 提高真实视觉抖动下的稳定性

建议：

- 优先微调 `D`
- 若接近中心后还是频繁清零，再看 `tolerance / stable_cycles`
- 不要急着加 `I`

---

## 13. 常见现象与处理建议

### 现象 1：怎么调都很慢

优先检查：

- `P` 是否太小
- `camera_aim_max_step` 是否太小

### 现象 2：快到中心时左右来回摆

优先检查：

- `P` 是否太大
- `D` 是否太小
- `max_step` 是否太大

### 现象 3：已经很接近，但老是触发不了 `capture_ready`

优先检查：

- `tolerance` 是否太小
- `stable_cycles` 是否太大
- 视觉误差是否本身就在抖

### 现象 4：动作很僵，不积极修正

优先检查：

- `D` 是否太大
- `max_step` 是否太小

### 现象 5：刚切目标时动作异常

当前代码已经在“目标进入 `TRACKING`”时重置：

- `aim_close_count_`
- PID 内部状态

如果仍出现异常，应继续排查：

- 视觉端目标 `err_x / err_y` 是否突变
- 当前目标是否被错误复用
- `label_instance_id` 是否稳定

---

## 14. 当前阶段的推荐结论

对你现在这版 `camera_aim`，推荐的默认调参原则是：

- 先不动 `I`
- 先用较小 `max_step`
- 先用中等 `P`
- 再用少量 `D` 去抑制接近中心时的摆动
- 闭环跑通后，再收紧 `tolerance` 和 `stable_cycles`

一句话总结：

**先求稳，再求快；先让它稳定对准，再让它更漂亮地对准。**

# 坐标系与航向控制交付说明

## 1. 适用范围

本文档总结当前工程中任务 YAML 的 `frame` 与 `yaw` 用法，面向队友交接与联调使用。

本文基于当前代码实现，不是泛化的 ROS2/PX4 理论说明。实际语义以如下实现为准：

- `frame` 解析：`src/drone_control/src/mission_controller_node.cpp`
- `world_body` 建系：`src/drone_control/src/tf_bridge_node.cpp`
- `move` 执行：`src/drone_control/include/drone_control/action_executor.hpp`

## 2. 先记住两个大前提

### 2.1 外部坐标统一按 ENU 理解

当前任务里涉及的位置坐标，外部统一按 ENU 理解：

- `x`：East，东
- `y`：North，北
- `z`：Up，上

注意：

- MAVROS 常用 ENU
- PX4 内部常用 NED
- 写任务 YAML 时不要把这两套坐标混用

### 2.2 当前工程里 `frame` 只有 3 种有效写法

任务 YAML 里推荐只使用以下 3 个值：

- `body`
- `world_enu`
- `world_body`

代码解析规则：

- `body` -> `BODY`
- `world_enu` -> `WORLD_ENU`
- 其他任意字符串 -> 默认按 `WORLD_BODY` 处理

注意：

- 虽然未知字符串会被解析成 `WORLD_BODY`
- 但 `header.frame_id` 仍保留原字符串
- 后续 TF 变换可能直接失败

因此不要写错字，不要自造 frame 名称。

## 3. 三个坐标系的作用和区别

### 3.1 对照表

| `frame` | 原点 | 轴方向 | 是否随当前机头实时变化 | 典型用途 |
| --- | --- | --- | --- | --- |
| `body` | 当前动作参考点附近，执行时按当前机体方向解释位移 | 当前机头方向 | 会 | 相对当前机头前后左右移动 |
| `world_enu` | 固定世界原点 | 固定 ENU | 不会 | 地图绝对坐标、场地固定点 |
| `world_body` | 起飞/初始化时无人机所在位置 | 起飞/初始化时无人机朝向 | 不会 | 相对起飞点写任务、比赛模板复用 |

### 3.2 `body` 的语义

`body` 表示当前机体系。

可以把它理解成：

- `x+` 沿当前机头前方
- `y+` 沿当前机体左/右侧方向中的固定定义方向
- `z+` 向上

在执行器里，`body` 会先把位移向量按当前姿态转换到 ENU，再叠加到当前位置附近。

适合场景：

- 想让无人机沿当前机头方向前进
- 转完向以后，再沿新的机头方向继续移动

不适合场景：

- 想让同一份任务在不同起飞朝向下仍保持完全相同的场地轨迹

### 3.3 `world_enu` 的语义

`world_enu` 表示固定世界坐标系。

特点：

- 原点固定
- 轴方向固定
- 不受无人机起飞朝向影响

适合场景：

- 地图上已有绝对坐标点
- 仿真场景里目标点位置固定
- 多次飞行都要飞到同一个世界点

进一步说，`world_enu` 最适合这类任务：

- 已经知道场地中的目标点绝对坐标
- 任务目标来自 GPS、视觉定位、地图标定结果或外部定位系统
- 希望无人机不管起飞朝向如何变化，都飞向同一个物理位置
- 希望多次测试复现完全一致的场地轨迹

不太适合这类任务：

- 只知道“相对起飞点前方多少米”，不知道世界绝对点位
- 比赛现场每次摆机位置变化较大，但希望直接复用同一份任务
- 更关心“沿当前机头前进多少米”这种相对动作

例子：

```yaml
- type: "move"
  frame: "world_enu"
  position: [2.0, 0.0, 1.2]
  yaw: 0.0
```

含义：

- 飞到世界坐标 `(2.0, 0.0, 1.2)`
- 无论起飞时机头朝东还是朝北，这个目标点都不变

这里可以把 `world_enu` 理解成“地图坐标”：

- `x` 沿东向
- `y` 沿北向
- `z` 向上

因此：

- `position: [2.0, 0.0, 1.2]` 表示向东 2 米、高度 1.2 米
- `position: [0.0, 3.0, 1.2]` 表示向北 3 米、高度 1.2 米

### 3.3.1 `world_enu` 的具体用法

任务 YAML 中的典型写法如下：

```yaml
- type: "move"
  frame: "world_enu"
  position: [2.0, 1.0, 1.2]
  yaw: 0.0
  tolerance: 0.12
```

字段含义：

- `frame: "world_enu"`：目标点按固定世界坐标解释
- `position: [x, y, z]`：目标位置，单位是米
- `yaw`：目标航向，单位是弧度
- `tolerance`：到点判定的距离容差

在当前工程里的执行逻辑是：

- `frame: "world_enu"` 会被解析为 `WORLD_ENU`
- 执行器会直接按世界坐标目标下发位置设定值
- 不会像 `world_body` 那样先做“起飞参考系 -> 世界系”的转换

因此只要世界坐标本身定义正确，`world_enu` 的目标点就具有稳定、明确、可复现的意义。

### 3.3.2 `world_enu` 和 `yaw` 的关系

在 `world_enu` 下：

- `position` 决定无人机要飞到场地里的哪个固定点
- `yaw` 决定无人机到达该点时希望朝向哪里

这两个量是相关但独立的：

- 修改 `position`，会改变目标位置
- 修改 `yaw`，不会改变 `world_enu` 坐标系本身
- 无人机转向后，`world_enu` 的轴方向仍然保持固定不变

例如：

```yaml
- type: "move"
  frame: "world_enu"
  position: [2.0, 1.0, 1.2]
  yaw: 1.57
```

表示的是：

- 飞到世界坐标 `(2.0, 1.0, 1.2)`
- 到点后朝向 `90 deg`
- 不是“坐标轴也跟着旋转了 90 deg”

### 3.3.3 `world_enu` 的使用建议

- 如果任务点来自地图、标定或绝对定位系统，优先选 `world_enu`
- 如果要做固定航线复现实验，优先选 `world_enu`
- 如果当前任务本质是“相对起飞点”，不要勉强用 `world_enu`
- 真机使用前，要先确认定位源稳定，否则绝对点位会整体漂移

### 3.4 `world_body` 的语义

`world_body` 不是“当前机体系”，而是“起飞/初始化时冻结下来的机体系”。

当前工程里，系统会在初始化阶段取一个 `start_pose`，然后建立一条静态 TF：

- 父坐标：`world_enu`
- 子坐标：`world_body`

其含义是：

- `world_body` 原点 = 起飞/初始化时无人机位置
- `world_body` 轴方向 = 起飞/初始化时无人机姿态方向

一旦这条 TF 建好，后续不会随着无人机转向而更新。

适合场景：

- 任务想按“相对起飞点”来写
- 比赛现场每次摆机位置有变化，但任务逻辑想保持一致
- 希望“前方 1 米”永远表示起飞时机头前方的 1 米

不适合场景：

- 希望转向后，后续坐标轴也跟着当前机头旋转

## 4. `world_body` 和 `world_enu` 的核心区别

最重要的区别不是都叫“world”，而是参考基准不同。

### 4.1 `world_enu`

- 参考的是固定世界
- 轴方向永远不变
- 适合描述场地绝对点位

### 4.2 `world_body`

- 参考的是起飞/初始化时的无人机
- 原点是起飞点
- 轴方向是起飞时机头方向
- 建好以后保持不变

### 4.3 一个最直观的例子

假设起飞时机头朝东。

此时：

- `world_body` 的 `x+` 指向东
- `world_body` 的 `y+` 指向与起飞姿态对应的横向方向

若此后无人机飞到某点并原地转向 90 度，机头改为朝北：

- `world_body` 坐标系不会跟着变
- `world_body` 的 `x+` 仍然指向起飞时定义的东

所以这条指令：

```yaml
- type: "move"
  frame: "world_body"
  position: [3.0, 0.0, 1.2]
  yaw: 1.57
```

表示的是：

- 去 `world_body` 里的 `(3.0, 0.0, 1.2)`
- 即相对起飞参考系向前 3 米的位置
- 不是“沿当前机头再前进 3 米”

如果想表达“沿当前机头再往前飞”，应该考虑 `body`，不是 `world_body`。

## 5. `world_body` 的推荐用法

### 5.1 最常见用法

`world_body` 最适合写“相对起飞点”的任务模板。

例如正方形航线：

```yaml
- type: "move"
  frame: "world_body"
  position: [1.0, 0.0, 1.2]
  yaw: 0.0

- type: "move"
  frame: "world_body"
  position: [1.0, 1.0, 1.2]
  yaw: 1.57

- type: "move"
  frame: "world_body"
  position: [0.0, 1.0, 1.2]
  yaw: 3.14

- type: "move"
  frame: "world_body"
  position: [0.0, 0.0, 1.2]
  yaw: -1.57
```

这类任务的优点是：

- 不要求你提前知道世界绝对坐标原点
- 每次都以起飞点为参考
- 比赛现场更容易复用

### 5.2 使用前提

要保证 `world_body` 好用，前提是：

- 起飞初始化时姿态稳定
- 建系时位置估计稳定
- TF 初始化已经完成

否则会出现：

- 起飞参考点漂移
- 起飞参考朝向不准
- `world_body -> world_enu` 变换失败

### 5.3 使用建议

- 起飞后先悬停几秒，再进入基于 `world_body` 的航线
- 同一套任务内，尽量统一使用一种主要参考系
- 如果队友主要按相对起飞点思考轨迹，优先用 `world_body`
- 如果队友主要按地图坐标思考轨迹，优先用 `world_enu`

## 6. 如何通过 `yaw` 改变航向

### 6.1 当前工程的控制方式

当前任务里，航向主要通过 `move` 动作里的 `yaw` 控制。

YAML 中：

```yaml
- type: "move"
  frame: "world_body"
  position: [1.0, 0.0, 1.2]
  yaw: 1.57
  tolerance: 0.12
```

代码会做两件事：

1. 读取 `yaw`
2. 用 `yaw` 生成目标四元数，写进 `PoseStamped`

然后执行器把这个带姿态的目标发到：

- `/mavros/setpoint_position/local`

### 6.2 `yaw` 的单位

`yaw` 使用弧度制，不是角度制。

常用值如下：

| 角度 | 弧度 |
| --- | --- |
| `0 deg` | `0.0` |
| `90 deg` | `1.57` |
| `180 deg` | `3.14` |
| `-90 deg` | `-1.57` |

### 6.3 原地转向的写法

如果想“先飞到点，再原地转向”，做法是：

- 连续写两个 `move`
- 两个动作的位置相同
- 第二个动作只修改 `yaw`

例如：

```yaml
- type: "move"
  frame: "world_body"
  position: [2.0, 0.0, 1.2]
  yaw: 0.0
  tolerance: 0.12

- type: "move"
  frame: "world_body"
  position: [2.0, 0.0, 1.2]
  yaw: 1.57
  tolerance: 0.12
```

这表示：

- 第一个动作先飞到目标点
- 第二个动作在同一点把航向转到 `90 deg`

### 6.4 为什么建议转向后加 `hover`

当前工程里，`move` 动作完成条件只检查位置误差，不检查航向误差。

这意味着：

- 位置到了，动作就可能判定完成
- 此时航向不一定已经完全转到位

因此建议在原地转向动作后补一个短暂悬停：

```yaml
- type: "move"
  frame: "world_body"
  position: [2.0, 0.0, 1.2]
  yaw: 1.57
  tolerance: 0.12

- type: "hover"
  duration: 2.0
```

这样更利于：

- 等待偏航收敛
- 便于观察转向效果
- 防止过早进入下一个动作

## 7. 一个常见误区

### 误区：`world_body` 会随着当前机头变化

这是错误理解。

正确理解是：

- `body` 才和当前机头实时相关
- `world_body` 只和起飞/初始化时的姿态相关
- 起飞后再怎么转向，`world_body` 的轴都不会跟着实时旋转

因此：

- 想写“起飞参考系任务模板”用 `world_body`
- 想写“当前机头前进 1 米”用 `body`

## 8. 推荐给队友的选型规则

如果队友在写任务前不确定选哪个坐标系，可以直接按下面判断：

### 8.1 选 `world_enu`

满足任一条即可：

- 已知固定地图坐标
- 希望不同起飞朝向下仍飞向同一个场地点
- 任务目标来自绝对定位系统

### 8.2 选 `world_body`

满足任一条即可：

- 任务想相对起飞点描述
- 比赛临场更关心“前后左右多少米”
- 同一份任务需要在不同摆机位置下反复复用

### 8.3 选 `body`

满足任一条即可：

- 希望始终沿当前机头方向运动
- 转向后，后续位移也要跟着新机头方向变化
- 做局部机动或近距离相对移动

## 9. 调试与联调建议

### 9.1 先检查参考系是否统一

队友联调时，先确认这几个问题：

- 当前任务点是按 ENU 想的，还是按起飞参考系想的
- 起飞点是否需要作为任务原点
- 转向后，后续移动是否应该跟着当前机头变

### 9.2 先做小航线验证

建议先验证：

1. 原地起飞
2. `world_body` 下飞到 `(1, 0, 1.2)`
3. 原地转 `90 deg`
4. 再飞到 `(1, 1, 1.2)`

这样最容易看出：

- `world_body` 是否理解正确
- `yaw` 转向是否生效
- 转向后坐标轴是否被错误理解为“实时机体系”

### 9.3 真机前的安全检查

上真机前至少确认：

- RC 丢失/数传丢失 failsafe 已配置
- 低电量保护已配置
- 起飞点已明确
- 测试场地有足够安全余量

## 10. 一句话总结

- `world_enu`：固定世界坐标，适合绝对点位
- `world_body`：起飞时冻结的参考系，适合相对起飞点任务
- `body`：当前机体系，适合沿当前机头方向运动
- `yaw`：通过 `move` 动作设置，单位是弧度
- 想“到点后原地转向”，就写“同位置、不同 `yaw` 的第二个 `move`”，最好再补一个 `hover`
